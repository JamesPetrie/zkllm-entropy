# Security Audit: zkllm-entropy Proof System

## Soundness, Correctness, and Zero-Knowledge Assessment

**Date:** 2026-03-28
**Scope:** Full codebase and protocol review of https://github.com/JamesPetrie/zkllm-entropy
**Methodology:** Static code analysis, mathematical verification against literature, red-team attack construction

---

## Executive Summary

The zkllm-entropy system aims to prove that an LLM's output has bounded covert channel capacity by proving conditional entropy bounds over committed model weights. The system uses the Goldilocks field (p = 2^64 − 2^32 + 1) with FRI polynomial commitments, sumcheck protocols, and LogUp table lookups.

**Bottom line:** The low-level cryptographic primitives (field arithmetic, NTT, Merkle trees, FRI folding, sumcheck polynomial construction) are correctly implemented. However, there are **critical soundness gaps at the integration layer** that would allow a dishonest prover to forge arbitrarily low entropy bounds. The system also **does not achieve zero-knowledge** in its current form. Additionally, the protocol as currently deployed has **no interactive challenge mechanism** — the prover generates its own challenges, which voids the soundness guarantees of all sumcheck and FRI protocols.

### Severity Summary

| Severity | Count | Summary |
|----------|-------|---------|
| **Critical** | 3 | Prover self-challenges, weight-binding proofs not serialized, total_win unproven |
| **High** | 4 | Argmax verification incomplete, no ZK property, FRI-PCS non-succinct, challenge pre-generation |
| **Medium** | 4 | Clamping inconsistency, multiplicity not range-proven, tLookup degree concerns, extension field not used |
| **Low** | 3 | Proof format versioning, NTT allocation overhead, parameter domain checks |

## Part I: Correctness of Primitives

### 1.1 Goldilocks Field Arithmetic — CORRECT

**File:** `src/field/goldilocks.cuh`

The field p = 2^64 − 2^32 + 1 arithmetic is correctly implemented using the Plonky2-style reduction. The key identity 2^64 ≡ 2^32 − 1 (mod p) is correctly exploited:

- **Multiplication:** Given a·b = (hi, lo) as a 128-bit product, the reduction computes `lo − (hi >> 32) + (hi & 0xFFFFFFFF) · (2^32 − 1)` with correct borrow/carry handling. Verified against the Plonky2 reference implementation.
- **Addition/Subtraction:** Correct conditional reduction with overflow/underflow checks.
- **Inverse:** Fermat's little theorem a^(p−2) mod p via binary exponentiation. Correct.

**Literature check:** The Goldilocks field is well-studied (Plonky2, Winterfell/STARK). The implementation matches known correct constructions.

### 1.2 NTT (Number Theoretic Transform) — CORRECT

**File:** `src/poly/ntt.cu`

- Uses generator g = 7 for the multiplicative group, computing ω_k = g^((p−1)/2^k).
- Cooley-Tukey radix-2 DIT with bit-reversal permutation.
- Inverse NTT correctly uses ω^(−1) and scales by 1/n.
- Coset NTT correctly multiplies by shift factors before/after transform.
- The Goldilocks field supports 2^32-th roots of unity (since p − 1 = 2^32 · (2^32 − 1)), which is ample.

### 1.3 Merkle Trees — CORRECT

**File:** `src/commit/merkle.cu`

- SHA-256 compression function is standard and correct (verified constants, round structure).
- Tree construction: leaves at indices [0, n), parents computed bottom-up to root at index 2n−2.
- Proof generation: collects sibling hashes from leaf to root.
- Verification: recomputes hash path from leaf to root, correctly handles left/right child ordering via `idx & 1`.
- Deterministic: same data always produces same root (verified in tests).

### 1.4 Polynomial Operations — CORRECT

**File:** `src/poly/polynomial.cu`

- Horner's method evaluation is standard and correct.
- The multilinear extension eq(u,v) = 2uv − u − v + 1 is the correct formula.
- Arithmetic operations (add, sub, mul, div) are correct for the Goldilocks branch.

### 1.5 FRI Folding — CORRECT

**File:** `src/commit/fri.cu`

The FRI folding formula:

```
f_{r+1}(y) = (f_r(x) + f_r(−x))/2 + α · (f_r(x) − f_r(−x))/(2x)
```

where y = x², is correctly implemented. The domain offset squaring (`offset ← offset²`) at each fold is correct. Query position tracking through folding layers is correct. The verification routine correctly recomputes folded values and checks Merkle proofs at each layer.

**Literature check:** Matches the standard FRI construction from Ben-Sasson et al. (ICALP 2018) and the ethSTARK documentation.

## Part II: Critical Soundness Failures

### C1: PROVER GENERATES OWN CHALLENGES — CRITICAL

**Location:** Throughout `zkllm_entropy.cu`, `zkentropy.cu`, `proof.cu`

**The Problem:** All challenge vectors in the proof system are generated by the prover using `random_vec()`, which calls `curand` on the GPU. This is equivalent to the prover choosing its own challenges, which **completely destroys the soundness of every sumcheck, FRI query, and lookup argument in the system**.

The sumcheck protocol's soundness guarantee is: a cheating prover who knows all challenges in advance can construct false proofs that pass verification with probability 1 (not d/|F|). The Schwartz-Zippel lemma only provides soundness when the evaluation point is unknown to the polynomial's constructor at the time of construction.

**Reference:** Lund, Fortnow, Karloff, Nisan (1992): "The verifier's random choices must be hidden from the prover until after the prover commits each round polynomial." Thaler's survey ("The Unreasonable Power of the Sum-Check Protocol") explicitly states that if the prover knows all challenges before the protocol begins, it can always cheat.

**Attack:** A dishonest prover:
1. Chooses the challenges c₁, ..., cₙ it will use
2. Constructs false univariate polynomials p₁, ..., pₙ such that pᵢ(0) + pᵢ(1) = claimᵢ and pᵢ(cᵢ) = claimᵢ₊₁ for any desired false claims
3. The verification passes perfectly

**Fix Options:**
- **(A) True interactive protocol:** Verifier generates challenges after receiving each round polynomial. This is the design intent stated in `design-goals.md`. The `interactive_verifier.cpp` implements this but is not the primary verification path.
- **(B) Fiat-Shamir transform:** Derive challenges as H(transcript || round_polynomial) using a cryptographic hash. This makes challenges unpredictable without finding hash preimages. However, over a 64-bit field, grinding attacks become feasible (see M1 below). Would require proof-of-work or extension field challenges.

**Severity:** CRITICAL. This single issue voids every soundness claim in the system.

### C2: WEIGHT-BINDING PROOFS NOT SERIALIZED — CRITICAL

**Location:** `bin/zkllm_entropy.cu:196-210`

**The Problem:** The prover runs `zkFC.prove()` and `verifyWeightClaim()` to prove that logits are correctly computed from committed weights, but these proofs are never written to the proof file. The verifier receives only the entropy proof polynomials with no way to link them back to the committed model weights.

**Attack:** A dishonest prover:
1. Ignores the committed model weights entirely
2. Fabricates any logits tensor that produces low entropy (e.g., one-hot distributions)
3. Correctly proves the entropy computation over the fabricated logits
4. The verifier accepts, because it cannot check that the logits came from the committed weights

**Impact:** The entire causal chain — committed weights → logits → entropy — is broken. The prover can claim any entropy bound.

**Fix:** Serialize the zkFC sumcheck proof, the Hadamard product proof for RMSNorm, and the FRI commitment opening proofs into the proof file. The verifier must check:
1. The FRI commitment root matches the public weight commitment
2. The zkFC sumcheck reduces to a claim on the committed weight polynomial
3. The FRI opening proof validates that claim against the commitment

### C3: TOTAL WIN PROBABILITY IS SELF-REPORTED — CRITICAL

**Location:** `src/entropy/zkentropy.cu:474-488`, `python/verify_entropy.py:222-229`

**The Problem:** The `total_win` value (sum of all win probabilities for a token position) is included in the proof as a constant polynomial. The verifier checks only weak bounds: `win_prob ≤ total_win ≤ vocab_size × cdf_scale`.

**Attack:** A dishonest prover:
1. Computes honest win_prob = 1000 for the actual token
2. Claims total_win = 1001 (just barely above win_prob, satisfying the bound)
3. This yields q_idx ≈ 2^log_precision (near-certainty)
4. Surprise = −log₂(q_idx / 2^log_precision) ≈ 0 bits
5. Honest total_win might be 50,000, yielding surprise ≈ 5.6 bits

**Quantitative impact:** For a vocabulary of 32,000 tokens with cdf_scale = 32,768:
- Honest total_win ≈ 32,768 (since CDF values approximately sum to 1 × scale)
- Attack total_win = win_prob + 1 → surprise drops from ~5–15 bits to ~0 bits per position
- Over 1,024 positions, total entropy drops from ~3,300 bits to ~0 bits

**Fix:** The prover must prove `total_win = Σᵢ win_probs[i]` via a sumcheck over the vocabulary dimension. The infrastructure already exists (`inner_product_sumcheck_interactive` with a ones vector). The plan in `plan-remaining-proofs.md` describes this as "~20 lines of code." This is the highest-impact fix relative to effort.

## Part III: High-Severity Soundness Issues

### H1: ARGMAX VERIFICATION IS INCOMPLETE — HIGH

**Location:** `src/zknn/zkargmax.cu`, `python/verify_entropy.py:204-209`

**The Problem:** The argmax proof generates 8 polynomials per position encoding bit-decomposition of differences, indicator constraints, and binary checks. The verifier reads only the constant terms of `ind_sum` (must equal 1) and `ind_dot` (must equal 0), without verifying the sumcheck rounds that prove these constraints hold over the entire tensor.

**What's missing:** The verifier does not check:
- That the indicator vector is actually one-hot (binary sumcheck)
- That the bit decomposition correctly reconstructs the differences
- That each bit is actually binary (not an arbitrary field element)

**Attack:** A dishonest prover:
1. Claims a false argmax position t_star with a higher logit value
2. Constructs diffs that are all non-negative (hiding the true maximum)
3. Sets `ind_sum = 1` and `ind_dot = 0` as constants
4. The verifier accepts because it only checks these constants

**Impact:** The prover can choose which token appears to be the argmax, manipulating the diff values and thus the CDF lookups and entropy.

**Fix:** The verifier must replay the sumcheck rounds for the binary constraints and bit-decomposition reconstruction. This requires serializing the full sumcheck transcript (round polynomials) and checking `p(0) + p(1) = claim` at each round with verifier-chosen challenges.

### H2: NO ZERO-KNOWLEDGE PROPERTY — HIGH

**The Problem:** The system reveals per-position intermediate values in the clear: logit activations, difference values, win probabilities, total win, quantized probability indices, and per-token surprise values. Additionally, FRI has no polynomial blinding.

**What is revealed:**
- **Per-token surprise values:** Directly reveal the model's confidence in each output token
- **Difference values (d_actual):** Reveal the gap between the top logit and each token's logit
- **Win probabilities:** Reveal the Gaussian-convolved probability of each token being selected
- **FRI evaluations:** Reveal polynomial values at queried positions (no blinding)

**Why this matters:** The design goal states "Model weights hidden behind cryptographic commitment." But the per-token surprise values alone reveal the entropy structure of the model's output distribution — an adversary can reconstruct approximate logit rankings, identify which tokens the model is "uncertain" about, and potentially extract information about model behavior that should remain private.

**Literature:** Haböck and Al Kindi (ePrint 2024/1037, "A note on adding zero-knowledge to STARKs") describe the requirements for ZK in FRI-based systems: a blinding polynomial h(x) must be added such that f'(x) = f(x) + Z_D(x) · h(x), where Z_D is the vanishing polynomial of the trace domain. Without this, FRI oracles leak information about the committed polynomial.

**Fix options:**
- **(A) Commit all intermediate values:** Replace plaintext per-position values with Merkle commitments. The verifier learns only the aggregate entropy bound, not per-token values. This is architecturally feasible with existing modules.
- **(B) Add FRI blinding:** Before FRI commitment, add a random masking polynomial of sufficient degree. This prevents polynomial evaluations from revealing witness information.
- **(C) Hybrid:** If the threat model accepts revealing per-token surprise but not logit structure, selectively blind only weight-derived values.

**Note:** The design document acknowledges this and states zero-knowledge is "architecturally feasible but not yet implemented." This is an honest assessment. However, any claim that the system "hides model weights" must be qualified: while the raw weight values are not directly revealed, the per-token distribution information leaks significant structural information about the model.

### H3: FRI-PCS IS NON-SUCCINCT — HIGH

**Location:** `src/commit/fri_pcs.cu:61-78`

**The Problem:** The FRI polynomial commitment scheme's `open()` function recomputes the entire Merkle tree and evaluates the MLE directly from the committed data. This means the verifier must have access to the full committed data (all model weights) to verify an opening.

**Impact:** For a 7B-parameter model with 32-bit quantized weights, this means the verifier needs ~28 GB of data. For the intended 1T-parameter target, this would be ~4 TB. This is not practical for a lightweight verifier.

**What's expected:** A polynomial commitment scheme should allow the verifier to check an opening using only the commitment (Merkle root) and a logarithmic-size proof, not the full data. Standard FRI-based PCS achieves this via quotient polynomials: to prove f(u) = v, the prover constructs q(x) = (f(x) − v) / (x − u) and proves q is low-degree via FRI.

**Fix:** Implement the standard FRI-based evaluation proof:
1. Prover constructs quotient polynomial q(x) = (f(x) − v) / Πᵢ(xᵢ − uᵢ) for multilinear f
2. Prover commits q via FRI (Merkle tree + low-degree test)
3. Verifier checks: FRI accepts q, and the algebraic identity holds at a random point

This is standard in STARK implementations (ethSTARK, Winterfell, Plonky2).

### H4: HADAMARD SUMCHECK CHALLENGES PRE-GENERATED — HIGH

**Location:** `src/proof/proof.cu:283-284`

**The Problem:** In `hadamard_product_sumcheck_interactive()`, both challenge vectors u and v are generated upfront before the protocol starts:

```cuda
u_out = random_vec(num_rounds);
v_out = random_vec(num_rounds);
```

The folding challenges (v) should be generated one per round, after the prover commits each round polynomial. Generating them all upfront allows the prover to see all folding directions before committing any polynomials, which weakens soundness.

**Distinction:** The MLE evaluation point (u) can legitimately be chosen upfront (it defines which point to evaluate at). But the sumcheck folding challenges (v) must be sequential.

**Impact:** Reduced from critical because the prover also generates these challenges (see C1), so this is currently a secondary issue. However, once C1 is fixed (interactive or Fiat-Shamir), this pre-generation will become the binding constraint on soundness.

**Fix:** Generate each v[i] after receiving and committing the i-th round polynomial.

## Part IV: Medium-Severity Issues

### M1: 64-BIT FIELD WITHOUT EXTENSION — MEDIUM

**The Problem:** The Goldilocks field has 2^64 elements. Each application of Schwartz-Zippel over this field contributes soundness error d/2^64. For a protocol with v sumcheck variables of degree d, the total error is v·d/2^64.

For the entropy proof over 1,024 tokens with vocab_size 32,000:
- Sumcheck variables: ~15 per sub-proof × ~6 sub-proofs ≈ 90 variables
- Total soundness error: ~90/2^64 ≈ 5 × 10^−18

This is acceptable for interactive proofs. However:

**In the non-interactive (Fiat-Shamir) setting**, a cheating prover can grind: try many different first-round messages until the hash-derived challenge is favorable. Over a 64-bit field, each grinding attempt has a 1/2^64 chance of hitting a "lucky" challenge. With 2^32 grinding attempts (feasible in seconds on modern hardware), the effective soundness drops to ~90/2^32 ≈ 2 × 10^−8.

**Literature:** Plonky2 addresses this by using degree-2 extensions of Goldilocks (128-bit extension field) for verifier challenges while keeping witness polynomials over the base field. The Winterfell STARK library's security formula is: `security_bits = num_queries × log₂(blowup_factor) + grinding_factor`.

**Recommendation:** If the system moves to Fiat-Shamir (non-interactive), implement:
1. Extension field challenges (quadratic extension of Goldilocks)
2. Proof-of-work grinding factor (e.g., 16–32 bits)
3. Multiple FRI queries (currently 1 per opening)

For the interactive setting (design intent), 64-bit base field is adequate.

### M2: CLAMPING CREATES PROOF/WITNESS MISMATCH — MEDIUM

**Location:** `src/entropy/zkentropy.cu:282-286, 382-385, 504`

**The Problem:** Win probability values are clamped with `clamp_min_one_kernel()` after computation, but the unclamped raw values are used for the extraction proof (line 504: `actual_wp_raw`). This creates a mismatch between:
- The clamped values used in the quotient-remainder computation
- The unclamped values used in the inner product sumcheck for token extraction

If clamping changes any values (i.e., some win_prob values are 0 before clamping), the extraction proof proves a claim about different data than the quotient-remainder proof uses.

**Impact:** Could cause proof rejection for honest provers (completeness failure), or if not caught, could allow a dishonest prover to exploit the mismatch.

**Fix:** Use a single consistent tensor for all sub-proofs, either always-clamped or always-raw, and prove the clamping step itself if needed.

### M3: TLOOKUP POLYNOMIAL DEGREE CONCERN — MEDIUM

**Location:** `src/zknn/tlookup.cu:188-202`

**The Problem:** In Phase 1 of the tLookup protocol, the round polynomial is constructed as:

```cuda
Polynomial p0({temp0(u_), temp1(u_), temp2(u_)});  // degree-2
p0 *= Polynomial::eq(u.back());                      // eq has degree 1
// Result: degree 3
```

The sumcheck protocol assumes round polynomials have degree at most d (typically 2 for multilinear). A degree-3 polynomial evaluated at only 3 points (0, 1, challenge) cannot be uniquely determined — it requires 4 points.

**Impact:** If the verifier interpolates the round polynomial from 3 points assuming degree 2, but the actual polynomial has degree 3, the interpolation is incorrect and the check `p(0) + p(1) = claim` may pass for a false polynomial.

**Uncertain:** I am not fully certain this is exploitable because the polynomial construction may implicitly guarantee degree ≤ 2 through algebraic cancellation. This requires more careful analysis of the specific polynomial structure in the LogUp argument.

**Recommendation:** Verify that the polynomial `p0 * eq(u.back()) + p1 + C * TWO_INV` has degree ≤ 2 in the sumcheck variable, or provide 4 evaluation points per round if degree 3.

### M4: MULTIPLICITY VECTOR NOT RANGE-PROVEN — MEDIUM

**Location:** `src/zknn/tlookup.cu:590`

**The Problem:** The multiplicity vector m (counting how many times each table entry is looked up) is computed by the prover via `atomicAdd` on the GPU. The LogUp argument proves that the multiplicities are *consistent* with the lookup, but does not separately prove that each m[i] ≥ 0.

**Literature:** The LogUp argument (Haböck, ePrint 2022/1530) has a known issue with multiplicity overflow: if a table entry's multiplicity equals the field characteristic |F|, the contribution cancels to zero, making incorrect lookups invisible. For the Goldilocks field, this requires 2^64 lookups of a single table entry, which is infeasible in practice for this system (vocab_size ≤ 32,000).

**Recommendation:** Low risk in practice due to the enormous multiplicity required. Document the bound: total lookups per table entry must be < p = 2^64 − 2^32 + 1.

## Part V: Red Team Attack Scenarios

### Attack 1: Complete Entropy Forgery (exploits C1 + C2 + C3)

**Threat model:** Dishonest prover with access to committed model weights, wants to prove entropy = 0 for arbitrary output.

**Steps:**
1. Prover generates challenges c₁, ..., cₙ (C1: self-chosen)
2. Fabricates logits tensor where every position has the actual output token as a near-certainty:
   - logit[t_star] = 2^30, all others = 0
   - This makes win_prob[t_star] ≈ cdf_scale, all others ≈ 0
3. Sets total_win = win_prob[t_star] (C3: self-reported, passes bounds check)
4. q_idx = 2^log_precision (maximum), surprise = 0
5. Constructs valid sumcheck polynomials for the false claims (C1: knows challenges)
6. Skips weight-binding proof (C2: not serialized)
7. Total entropy = 0 bits for all 1,024 positions

**Result:** Verifier accepts a proof claiming 0 bits of entropy for output that actually has ~3,300 bits of entropy. The prover can embed arbitrary covert messages while appearing compliant.

**Feasibility:** Trivial. Requires only modifying the prover code, no computational effort.

### Attack 2: Selective Token Manipulation (exploits H1 + C3)

**Threat model:** Dishonest prover who wants to hide entropy at specific positions (e.g., tokens containing steganographic payload) while appearing honest elsewhere.

**Steps:**
1. For positions carrying covert data (e.g., 10% of tokens):
   - Claim a false argmax that happens to match the actual output token
   - This makes diff[actual_token] = 0, win_prob = cdf_scale
   - Set total_win = cdf_scale + 1
   - Surprise ≈ 0 bits
2. For remaining positions: compute honestly
3. Result: 90% of entropy is honest (~3 bits/token), 10% is forged to 0

**Impact:** Average entropy drops from ~3.2 bits/token to ~2.9 bits/token. The covert channel capacity at the forged positions is hidden.

**Feasibility:** Easy once C1 is exploited.

### Attack 3: Weight Substitution (exploits C2)

**Threat model:** Dishonest prover who committed to one model but runs a different (steganography-enabled) model.

**Steps:**
1. Commit to legitimate model weights W_honest
2. At inference time, run a modified model W_stego that embeds covert messages
3. Generate entropy proof using W_stego's actual outputs and logits
4. The proof is arithmetically correct (CDF, log, entropy all check out)
5. Weight-binding proof is not included, so verifier cannot detect W_stego ≠ W_honest

**Impact:** The entire verification framework is bypassed. The prover can run any model and prove any entropy bound for its outputs.

### Attack 4: Grinding Attack on Fiat-Shamir (if C1 is fixed via option B)

**Threat model:** Prover facing a Fiat-Shamir-transformed protocol over 64-bit field.

**Steps:**
1. Choose a false claim (e.g., total_win = 1000 instead of 50,000)
2. In round 1 of the sumcheck, construct the false polynomial
3. Compute challenge c₁ = H(transcript || p₁)
4. With probability d/2^64, the false polynomial passes at c₁
5. If it doesn't, modify the proof nonce and retry from step 2
6. Expected grinding cost: 2^64/d ≈ 2^63 hash evaluations per round

**Impact:** With a 64-bit field, each sumcheck round requires ~2^63 grinding attempts. This is infeasible for a single round but becomes marginal if the protocol has structural weaknesses that reduce the effective degree.

**Mitigation:** Extension field challenges (128-bit) make grinding cost 2^127 per round — completely infeasible.

### Attack 5: Distribution Inference from Proof (exploits H2)

**Threat model:** Curious verifier who wants to learn about the model's behavior.

**Steps:**
1. Read per-position surprise values from the proof file
2. Positions with surprise = 0: model's argmax matches output (high confidence)
3. Positions with surprise = 15 bits: model assigns near-zero probability to output
4. Reconstruct approximate per-token confidence profile
5. Use diff_actual values to estimate logit gaps
6. Aggregate across many proofs to build a statistical model of the LLM's behavior

**Impact:** Reveals the model's confidence distribution, which tokens it finds surprising, and approximate logit structure — all of which should be hidden if zero-knowledge is a goal.

## Part VI: Comparison with Literature

### vs. Original zkLLM (Sun et al., CCS 2024)

| Aspect | zkLLM (CCS 2024) | zkllm-entropy |
|--------|-------------------|---------------|
| **Field** | BLS12-381 (255-bit) | Goldilocks (64-bit) |
| **Commitment** | Hyrax/Pedersen (DLOG) | FRI/Merkle (hash) |
| **Trusted setup** | Required | Not required |
| **Post-quantum** | No (DLOG) | Yes (hash-based) |
| **Zero-knowledge** | Yes (Pedersen hiding) | No (no blinding) |
| **Verifier** | Succinct | Non-succinct (needs full data) |
| **Soundness** | Complete (all proofs verified) | Incomplete (S1, S2, S3 gaps) |
| **What's proven** | Correct inference | Entropy bound (not full inference) |
| **Performance** | ~15 min for 13B | ~7 min for 7B |

**Key insight:** zkllm-entropy trades zero-knowledge and succinctness for post-quantum security and no trusted setup. This is a valid design choice, but the soundness gaps (C1–C3) are implementation issues, not inherent to the design.

### vs. FRI Soundness Literature

The system uses blowup factor 2 and 1 query per opening. Per the conjectured FRI soundness bound:

```
security_bits = num_queries × log₂(blowup_factor) = 1 × 1 = 1 bit
```

This provides only **1 bit of security per FRI opening** in the non-interactive setting. In the interactive setting (design intent), the verifier's challenge provides the security — the FRI is used only for commitment binding, not for soundness against a computationally bounded adversary.

**Recommendation:** If moving to non-interactive, increase blowup factor to 4–8 and use 20–40 queries for 80–128 bits of conjectured FRI security.

### vs. Sumcheck Soundness Bounds

For the interactive protocol over 64-bit field with multilinear polynomials (degree 1 per variable):

```
soundness_error ≤ v/|F| = 90/2^64 ≈ 5 × 10^−18
```

This is excellent — far below any practical attack threshold. The soundness is adequate **if and only if** challenges are truly random and unknown to the prover (i.e., C1 must be fixed).

### vs. Steganography Bounds (Cachin 1998, Minimum Entropy Coupling 2023)

The protocol's entropy bound is mathematically sound as a covert capacity bound by Gibbs' inequality:

```
H(O | W, x) ≤ −Σᵢ log₂ q̂(oᵢ | W, x, o_{<i})
```

This holds for ANY approximate distribution q̂, with equality iff q̂ equals the true conditional distribution. The use of Gaussian convolution (σ_eff) to handle hardware nondeterminism is a sound approach — it inflates the entropy bound but remains a valid upper bound.

**Key theoretical result (from the paper in inference-verification):** The covert message capacity is bounded by the conditional entropy, regardless of the steganographic scheme used. This follows from the data processing inequality and is well-established in information theory.

**No issues found** with the theoretical entropy bounding framework. The problems are entirely in the proof system's implementation, not in the information-theoretic arguments.

## Part VII: What Is Solid

To be fair to the system, the following components are correctly implemented and mathematically sound:

1. **Goldilocks field arithmetic** — Matches Plonky2 reference, all 36 tests pass
2. **NTT** — Correct Cooley-Tukey with proper root computation, 6 tests pass
3. **Merkle trees** — Standard SHA-256, correct proof generation and verification, 9 tests pass
4. **FRI folding** — Mathematically correct domain transformations, 6 tests pass
5. **Sumcheck polynomial construction** — The round polynomials `p(x) = a₀b₀ + (a₁−a₀)b₀x + a₀(b₁−b₀)x + (a₁−a₀)(b₁−b₀)x²` are correct
6. **LogUp/tLookup argument structure** — The two-phase protocol is a correct implementation of the LogUp approach
7. **CDF and log table construction** — Correct numerical computation of Gaussian CDF and log₂ values
8. **Entropy bounding theory** — The Gibbs inequality approach to bounding covert channel capacity is information-theoretically sound
9. **Quantization analysis** — The 32-bit fixed-point representation with 21.2 bits of headroom is well-analyzed and safe for the 7B model

## Part IX: Areas of Uncertainty

The following points require further analysis that I was unable to fully resolve:

1. **tLookup Phase 1 polynomial degree (M3):** The multiplication of a degree-2 polynomial by `eq(u.back())` produces degree 3 in the sumcheck variable. Whether the combined polynomial `p0 * eq + p1 + C/2` actually has degree ≤ 2 through algebraic cancellation depends on the specific structure of the LogUp identity. I was unable to fully verify this without a more detailed pen-and-paper analysis of the rational function decomposition.

2. **LogUp claim construction:** The initial claim `claim = α + α²` in the tLookup interactive proof appears to be a specific algebraic identity related to the LogUp accumulator, but the derivation is not documented in the code. I believe this is: C = α·(Σ 1/(S+β)) − (Σ m/(T+β)) = 0 for correct lookups, and the claim α + α² encodes this via a change of variables, but I am not certain.

3. **Completeness of quotient-remainder proof (zkentropy.cu:518-565):** The quotient-remainder proof that `q·tw + r = wp·2^p` uses the same evaluation point `u_qr` for both the arithmetic check and the bit-decomposition range proofs. Whether this coupling introduces a completeness or soundness issue depends on whether the combined polynomial identity uniquely constrains the witness. I lean toward "probably fine" but am not certain.

4. **FRI-PCS binding strength:** The current FRI-PCS recomputes the Merkle tree during opening (rather than using the committed root). This means binding relies on the prover not changing the data between commitment and opening, enforced only by program control flow rather than cryptographic binding. In the honest implementation this is fine, but a modified prover could commit to one dataset and open another. This is related to C2 and H3.
