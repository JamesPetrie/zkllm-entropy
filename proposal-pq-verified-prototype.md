# Project Proposal: Formally Verified CPU Prototype for Post-Quantum ZK Matrix Multiplication

## Summary

We propose to build a formally verified, CPU-based prototype of a post-quantum zero-knowledge proof for matrix multiplication, using FRI-based polynomial commitments and a ZK sumcheck protocol. The prototype has three deliverables: (1) a Python reference implementation for correctness testing, (2) a Rust implementation structured for future GPU porting, and (3) a Lean4 specification of the verifier with a machine-checked soundness proof. Together, these address the primary trust gap in the current `zkllm-entropy` system: the verifier does not cryptographically check any sumcheck, FRI opening, or commitment binding.

---

## Motivation

### The current soundness gap

The `zkllm-entropy` README is explicit:

> *"Verifier is arithmetic-only. `verify_entropy.py` recomputes CDF/log/quantization from scalar proof values and checks consistency. It does not verify any sumcheck, commitment opening, tLookup proof, or argmax range proof. A full cryptographic verifier has not been implemented."*

This means the current system can detect an honest prover who made a computational error, but it **cannot detect a malicious prover** who submits plausible-looking scalar values that do not correspond to any valid proof. For the applications listed in the README — regulatory audits, verifying the model served matches the model purchased, bounding covert channels — this gap is critical. A dishonest prover could claim any entropy bound they wish.

### Why a formally verified prototype addresses this

The GPU CUDA codebase is difficult to audit formally: CUDA kernels are inherently hard to reason about, the proof serialization is incomplete, and the verifier is a Python script. Rather than attempting to formally verify the CUDA implementation directly, we build a parallel verified stack at smaller scale:

- The CPU prototype implements the same protocol as the GPU system, over the same Goldilocks field, with the same sumcheck and FRI structure — just for a single matrix multiplication rather than a full LLM layer.
- The Lean4 verified verifier proves that if the verifier accepts, the statement is true (up to a negligible soundness error, assuming FRI soundness).
- Once the prototype is verified correct, the GPU system can be validated by cross-testing: proofs generated by the GPU prover must be accepted by the CPU verifier, and vice versa.

### Why matrix multiplication specifically

Every linear layer in the LLM pipeline reduces to matrix multiplication. The `zkFC` module in `src/zknn/zkfc.cu` — which is currently missing from the proof file per the README — proves exactly this operation. Formally verifying the matrix multiplication proof is therefore the core case. All other layers (RMSNorm, lm_head, rescaling) either compose directly with matmul proofs or add a lookup argument on top.

## Goals

1. **Correctness:** Produce a sumcheck + FRI proof for a single matrix multiplication `C = A × B` that a clean-room verifier can check.

2. **Post-quantum security:** Use only hash-based commitments (Goldilocks field + SHA-256 Merkle trees over FRI codewords), consistent with the existing `goldilocks-fri` branch.

3. **Formal soundness:** Prove in Lean4 that the verifier rejects any false claim, assuming collision resistance of SHA-256 and soundness of FRI (stated as a named axiom with citations). The proof must be machine-checked.

4. **Cross-validation:** The Rust verifier must accept proofs generated by both the Python prover and the existing CUDA prover (once weight-binding serialization is complete), providing an independent correctness check on the GPU system.

5. **GPU-ready structure:** The Rust implementation must be structured so that the three computationally heavy operations — NTT, Merkle tree construction, and sumcheck round evaluation — can be replaced with CUDA kernel calls without changing the protocol logic.

## Protocol: ZK Sumcheck + FRI for Single Matmul

### Statement

Given public Merkle commitments `root_A`, `root_B`, `root_C` to matrices A (m×k), B (k×n), C (m×n) over the Goldilocks field, prove that C = A × B.

### Core protocol (three steps)

**Step 1 — Sumcheck.** The prover reduces the matrix product identity to a scalar claim via the sumcheck protocol. For random verifier challenges `r_x ∈ F^{log m}`, `r_z ∈ F^{log n}`:

```
C̃(r_x, r_z) = ∑_{y ∈ {0,1}^{log k}} Ã(r_x, y) · B̃(y, r_z)
```

The prover runs `log k` rounds of the sumcheck, sending a masked degree-2 polynomial per round. The masking uses `ρ_i · Y·(Y-1)` (Type-1 masking) to ensure no information about Ã or B̃ leaks through the round messages.

**Step 2 — FRI evaluation proofs.** After the sumcheck, the verifier holds accumulated challenges `r_y` and needs evaluations `Ã(r_x, r_y)` and `B̃(r_y, r_z)`. The prover supplies BaseFold-style FRI evaluation arguments for both. Each argument commits to a folded sequence of codewords and opens `Q` random positions per fold level (Merkle paths). Type-2 masking (committing to `M̃ + b` with a separately proved blinding polynomial `b`) ensures the FRI query openings reveal nothing about the matrix entries.

**Step 3 — Consistency.** The verifier checks that `Ã(r_x, r_y) · B̃(r_y, r_z)` equals the final sumcheck claim, and that `C̃(r_x, r_z)` matches the FRI evaluation proof for `root_C`.

The protocol is **interactive**: the verifier sends each challenge freshly after receiving the prover's message for that round. Challenges are drawn uniformly from F by the verifier. No hash function is used to derive challenges — the verifier is a live participant throughout.

### Soundness error

For Goldilocks (`|F| = 2^64 - 2^32 + 1`), `log k = 12` rounds (k = 4096), degree `d = 2`:

- Sumcheck error: `12 · 2 / |F| ≈ 2^{-58}`
- FRI error: with an interactive verifier, each query position is chosen fresh after commitment, so a single query per fold level suffices. Over `2s = 24` fold levels at rate `ρ = 1/2`, the error is `(1/2)^{24} = 2^{-24}`. Running the query phase `Q` times independently brings this to `(1/2)^{24Q}`.
- Combined (Q = 3): `≈ 2^{-72}` (FRI-dominated)

The interactive setting is strictly stronger than non-interactive here: the prover cannot adaptively choose which positions to open because the query positions arrive after commitment. Non-interactive FRI requires ~50 parallel queries to reach 100-bit security; interactive FRI reaches the same with 5 sequential single-position queries.

For the prototype, smaller matrices (k = 64, 256) are sufficient — the soundness error shrinks with field size, not matrix size.

## Deliverable 1: Python Reference Implementation

**Purpose:** Fastest path to a correct protocol. Used as ground truth for testing the Rust implementation.

**Scope:**

```
pq_matmul/
  field.py          # Goldilocks field arithmetic (mod 2^64 - 2^32 + 1)
  mle.py            # Multilinear extension: table → evaluator
  sumcheck.py       # ZK sumcheck prover and verifier (Type-1 masking)
  fri.py            # FRI commit, fold, query phase, Merkle tree
  eval_argument.py  # BaseFold-style evaluation proof (Type-2 masking)
  prover.py         # Top-level: commit A, B, C → run sumcheck → FRI evals
  verifier.py       # Top-level: check all constraints
  tests/
    test_roundtrip.py   # Honest prover → verifier accepts
    test_soundness.py   # Corrupt A/B/transcript → verifier rejects
    test_masking.py     # Check round polys are not correlated with matrix entries
```

**Key implementation note:** The verifier is a live process that sends each challenge only after receiving the prover's message for that round. In the Python implementation this is a simple function-call alternation: `prover.send_round(i)` → `verifier.send_challenge(i)`. For testing and cross-validation, the full transcript (all prover messages and all verifier challenges) is recorded to a file after the interaction completes. Replaying a transcript for verification is a separate `verify_transcript` function that checks all constraints without re-running the interaction.

**Matrix sizes:** s = 4 (16×16) for unit tests; s = 6 (64×64) for integration tests. Both run in under one second on CPU.

## Deliverable 3: Lean4 Verified Verifier

**Purpose:** Machine-checked proof that the verifier only accepts true claims about matrix multiplication, assuming FRI soundness and SHA-256 collision resistance.

### What gets proved

**Theorem (Matmul Verifier Soundness).** Assuming:
- `fri_soundness`: if the FRI verifier accepts a codeword commitment, the committed function is within distance δ of a polynomial of degree at most d (cited to Ben-Sasson et al. 2018, 2020)
- `hash_collision_resistant`: SHA-256 has no efficiently computable collisions

Then: if `verify(root_A, root_B, root_C, transcript) = true`, the matrices committed to under `root_A`, `root_B`, `root_C` satisfy `C = A × B` at the verifier's random evaluation point, with probability ≥ `1 - s·d/|F|` over the verifier's uniformly random challenges.

This is cleaner to prove formally than the non-interactive case: challenges are modeled as uniform draws from F, with no random oracle assumption required. The Schwartz-Zippel bound applies directly to each round without any additional hypothesis about the hash function.

### Proof structure

The proof decomposes into four components, each independently checkable:

**Component 1: Merkle soundness** (easy, ~1 day)

```lean
axiom hash_collision_resistant : ∀ (x y : ByteArray), SHA256 x = SHA256 y → x = y

theorem merkle_soundness (root : Hash) (leaf : F) (idx : Nat) (path : List Hash)
    (h : verify_path root leaf idx path = true) :
    ∃ leaves : Fin N → F, merkle_root leaves = root ∧ leaves ⟨idx, _⟩ = leaf := by
  induction path with
  | nil  => exact ⟨fun _ => leaf, by simp [merkle_root], rfl⟩
  | cons sibling rest ih =>
    simp [verify_path] at h
    -- Each step: hash(sibling, child) = parent, uniquely determines child by collision resistance
    obtain ⟨leaves, hroot, hleaf⟩ := ih h.2
    exact ⟨leaves, by rw [← hroot]; exact hash_collision_resistant _ _ h.1, hleaf⟩
```

**Component 2: Sumcheck soundness** (medium, ~1–2 weeks)

The key lemma is Schwartz-Zippel, which is in Mathlib:

```lean
-- From Mathlib: a nonzero degree-d polynomial has at most d roots
#check Polynomial.card_roots_le_degree
-- ∀ (p : Polynomial F), p ≠ 0 → p.roots.card ≤ p.natDegree

-- Round soundness: if ĥᵢ ≠ hᵢ (prover lied), check fails unless r hits the difference
lemma round_check_exposes_lie
    (ĥ h : Polynomial F) (hneq : ĥ ≠ h) (hdeg : (ĥ - h).natDegree ≤ d) :
    (Finset.univ.filter fun r : F => ĥ.eval r = h.eval r).card ≤ d :=
  have hne : ĥ - h ≠ 0 := sub_ne_zero.mpr hneq
  Polynomial.card_roots_le_degree _ |>.trans (by simp [hdeg]) -- ~3 lines

-- Main sumcheck soundness theorem (induction over s rounds)
theorem sumcheck_sound (f : (Fin s → Bool) → F) (v : F)
    (transcript : Fin s → Polynomial F) (challenges : Fin s → F)
    (h_checks : ∀ i, (transcript i).eval 0 + (transcript i).eval 1 =
                  if i = 0 then v else (transcript (i-1)).eval (challenges (i-1)))
    (h_final : f_mle challenges = (transcript (s-1)).eval (challenges (s-1))) :
    ∑ x : Fin s → Bool, f x = v ∨
    ∃ i, (transcript i).eval (challenges i) =
         (honest_round_poly f (fun j => challenges j)).eval (challenges i) := by
  induction s with
  | zero => left; simpa using h_final
  | succ n ih => ...  -- applies round_check_exposes_lie at each step
```

The Type-1 masking (`ρ·Y·(Y-1)`) factors into this proof naturally: `ĥ_i(0) = h_i(0)` and `ĥ_i(1) = h_i(1)` because `Y·(Y-1)` vanishes at 0 and 1, so the sum check `ĥ_i(0) + ĥ_i(1) = σ_{i-1}` is unaffected by masking. This is a one-line `simp` in Lean4.

**Component 3: Arithmetization correctness** (easy, ~2–3 days)

The claim that `C = A × B` is equivalent to the sumcheck statement is a direct algebraic identity:

```lean
-- MLE of matrix product equals sum of MLE products
theorem matmul_mle_eq (A : Matrix (Fin m) (Fin k) F) (B : Matrix (Fin k) (Fin n) F)
    (rx : Fin (Nat.log2 m) → F) (rz : Fin (Nat.log2 n) → F) :
    mle (A * B) rx rz = ∑ y : Fin (Nat.log2 k) → Bool,
      mle A rx (fun i => (y i : F)) * mle B (fun i => (y i : F)) rz := by
  simp [mle, Matrix.mul_apply, Finset.sum_comm]  -- follows from linearity of MLE
```

This is essentially `Finset.sum_comm` applied to the multilinear extension definition.

**Component 4: FRI soundness** (axiomatized)

```lean
-- FRI soundness: stated as a named axiom with citations
-- Source: Ben-Sasson, Bentov, Horesh, Riabzev — STOC 2018
--         Ben-Sasson, Carmon, Ishai, Kopparty, Saraf — 2020
axiom fri_soundness
    (oracle : Domain → F) (root : Hash) (proof : FRIProof) (d : Nat) (δ : Rat)
    (h : verify_fri root proof d = true) :
    ∃ p : Polynomial F, p.natDegree ≤ d ∧
      (Finset.univ.filter fun x => oracle x ≠ p.eval x).card ≤ δ * Finset.univ.card
```

Axiomatizing FRI is honest: the FRI soundness theorem has a published proof, but no complete machine-checked version exists in Lean4 Mathlib as of this writing. Naming it as an axiom (rather than silently assuming it) makes the trust boundary explicit. Future work could replace this axiom with a full proof as the Lean4 crypto library matures.

### Composition

With all four components, the end-to-end theorem follows by straightforward composition:

```lean
theorem matmul_verifier_sound
    (A B C : Matrix (Fin n) (Fin n) F)
    (proof : MatmulProof)
    (h : verify proof (merkle_root A) (merkle_root B) (merkle_root C) = true) :
    C = A * B := by
  -- 1. FRI soundness → commitments are well-formed
  -- 2. Merkle soundness → committed values are the actual matrices
  -- 3. Sumcheck soundness (Schwartz-Zippel) → sumcheck claim holds
  -- 4. Arithmetization correctness → sumcheck claim = C = A*B
  ...
```

## What This Does Not Cover

- **Non-linear layers** (RMSNorm, softmax, entropy, lookups): these require separate gadgets and are out of scope. The prototype targets the `zkFC` matmul case only.
- **Batched matrix multiplication**: proving all weight matrices in a transformer layer requires either multiple independent proofs or a random-linear-combination batch sumcheck. Out of scope, but the single-matmul case generalizes straightforwardly.
- **Full FRI formal proof in Lean4**: replacing the `fri_soundness` axiom with a machine-checked proof is a significant research effort (estimated 3–6 months for an expert). Out of scope, but the axiom is named and cited so future work can slot it in.
- **Floating-point inference**: the prototype uses exact Goldilocks arithmetic. The connection to floating-point LLM inference (the core of zkllm-entropy) is handled by the Gibbs-inequality layer already in the system, which sits above the matmul proof.

## Relation to Known Soundness Gaps

The README lists several gaps. This project directly addresses two of them:

> *"Verifier is arithmetic-only."*

The Rust verifier built here checks sumcheck constraints, FRI folding consistency, and Merkle opening proofs. It is the cryptographic verifier the README identifies as missing.

> *"Weight-binding proofs not serialized."*

The prototype proves commitment to specific matrix entries via the FRI evaluation argument. Once the CUDA serialization is updated (a separate task), the Rust verifier can check that the logits in the entropy proof derive from the committed weight matrices.

The other gaps (binary sumcheck serialization, `cdf_precision` mismatch, argmax-CDF merge) are implementation bugs in the entropy layer and are orthogonal to this work.
